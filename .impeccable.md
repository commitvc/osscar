# .impeccable.md — OSS Growth Index Design Context

## Design Context

### Users
The OSS Growth Index serves a mixed but intentional audience: **developers and open source maintainers** are the primary users, coming to benchmark project growth, discover fast-moving ecosystems, and validate technical choices. A secondary but equally important audience is **founders, investors, and VCs** — the collaboration between Supabase and >commit means this is also a signal for the startup/investment community scouting high-growth OSS organizations. The interface must read as credible and data-authoritative to both groups simultaneously.

### Brand Personality
**Bold, precise, developer-native.** Three words: **Confident. Technical. Kinetic.**

The emotional goal is the feeling you get from opening a well-designed developer dashboard that's also visually alive — not a sterile table dump, but not marketing fluff either. It should feel like the Supabase brand met a hacker leaderboard: dark, high-contrast, with numbers that feel important and ranked positions that carry weight. Founders and investors should feel the prestige of appearing on it. Developers should feel it was built for them.

### Aesthetic Direction
- **Dark mode primary.** Background near-black with high-contrast foreground. Light mode is secondary/bonus.
- **Supabase-influenced base:** Clean layout, sharp typography, minimal chrome, purposeful green accents from the Supabase palette.
- **Layered with dev-tool energy:** Monospace type for numbers and identifiers, data-dense tables that don't feel cluttered, terminal-adjacent aesthetics for secondary UI (ranks, tiers, badges).
- **Typography:** Geist (already installed) for UI copy; Geist Mono for stats, rankings, growth rates, and any code-adjacent data. Numbers should feel precise and deliberate.
- **Color use:** Neutral dark base with a primary accent pulled toward Supabase green (`#3ECF8E` or similar). Chart colors should use the existing blue/purple OKLCH spectrum. Ranking tiers and growth badges can use color meaningfully.
- **Anti-references:** Avoid generic SaaS dashboard aesthetics (soft shadows, rounded cards everywhere, pastel charts). Avoid anything that looks like a startup landing page masquerading as data. No excessive whitespace that undermines the data density.

### Design Principles

1. **Data is the hero.** Every layout decision should make rankings, growth rates, and metrics easier to scan and compare. Chrome and decoration exist only to support comprehension.

2. **Developer credibility first.** The interface should feel native to someone who lives in VS Code and GitHub. Monospace for data, high contrast, precise spacing — not polished for its own sake, but precise because accuracy matters.

3. **Prestige through restraint.** The "index" format implies authority. Resist the urge to over-decorate. A stark ranked list with great typography communicates more status than an animated card carousel.

4. **Supabase DNA.** Lean into the Supabase brand: dark backgrounds, green as the primary accent, clean sans-serif layout. This is a Supabase co-product — it should feel like it lives in that family.

5. **Dual-audience legibility.** Investors and founders skim for signals (tier badges, rank deltas, org names). Developers dig into the methodology and numbers. The layout should support both reading modes: summary at a glance, depth on demand.

## Tech Stack Reference (for design decisions)
- **Framework:** Next.js 16 (App Router), React 19
- **Styling:** Tailwind CSS v4, shadcn base-nova, CSS variables in OKLCH
- **Components:** Base UI primitives, CVA for variants
- **Data viz:** Recharts
- **Fonts:** Geist (sans) + Geist Mono — both already configured as CSS vars
- **Icons:** Lucide React
- **Color system:** OKLCH semantic tokens (`--color-primary`, `--color-muted`, etc.) with dark mode via `.dark` class
